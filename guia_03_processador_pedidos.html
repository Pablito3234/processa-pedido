Processador de Pedidos — Multi-serviço

Roteiro prático para turmas de ADS · Docker Compose → Kubernetes (k3d) · Flask + RabbitMQ + Worker Python + Postgres + Painel Flask

Módulo 0 — Visão geral

Arquitetura da aplicação

Sistema de processamento de pedidos usando RabbitMQ como broker real com ACK/NACK — diferente do Redis List, o RabbitMQ garante entrega mesmo que o worker caia no meio do processamento.

Os 5 serviços
1. loja (Flask · porta 5000) — Recebe POST /order com itens do carrinho. Publica mensagem na exchange RabbitMQ pedidos. Retorna order_id imediatamente sem esperar o processamento.
2. rabbitmq (RabbitMQ 3-management · porta 15672) — Broker de mensagens com exchange pedidos e fila pedidos.queue. Interface web de gerenciamento em :15672.
3. db (Postgres 15) — Tabelas pedidos (id, cliente, total, status, criado_em) e itens_pedido (pedido_id, produto, qty, preco).
4. worker (Python · pika) — Consome pedidos.queue com ACK manual. Valida estoque (simulado), faz INSERT no Postgres, depois ACK. Se crashar antes do ACK, RabbitMQ reentrega automaticamente.
5. painel (Flask · porta 3000) — Lista pedidos com status e detalhes. Lê diretamente do Postgres.
Fluxo completo com ACK
SIMULAÇÃO DE COMPRA ────────────────────────────────────────────────────── [Cliente] ──POST /order {cliente, itens[]}──▶ [loja Flask] │ gera order_id = uuid │ publish exchange "pedidos" [RabbitMQ] │ ◀── {order_id, status: "queued"} PROCESSAMENTO COM ACK MANUAL ────────────────────────────────────────────────────── [RabbitMQ] ──deliver──▶ [worker Python/pika] │ valida estoque (simulado) │ ┌────────────────┴────────────────┐ válido inválido │ │ INSERT pedidos (status=OK) NACK → requeue=False [Postgres] (mensagem descartada) │ ACK() ← só após INSERT bem-sucedido │ [RabbitMQ] remove da fila PAINEL ────────────────────────────────────────────────────── [Usuário] ──GET /──▶ [painel Flask :3000] │ SELECT pedidos + itens_pedido [Postgres]
Redis List vs RabbitMQ
Redis List: simples, rápido, sem ACK nativo — se o worker crashar após BLPOP, a mensagem se perde.
RabbitMQ: ACK/NACK explícito — mensagem fica "unacked" até ACK. Se worker cair, RabbitMQ reentrega. Exchange → binding → queue permite roteamento flexível. Ideal para pedidos onde perda é inaceitável.

import os, time, json, uuid import pika, psycopg2 from flask import Flask, request, jsonify app = Flask(__name__) RABBIT_HOST = os.environ.get('RABBIT_HOST', 'rabbitmq') RABBIT_USER = os.environ.get('RABBIT_USER', 'guest') RABBIT_PASS = os.environ.get('RABBIT_PASS', 'guest') def get_db(): return psycopg2.connect( host=os.environ.get('DB_HOST','db'), dbname=os.environ.get('POSTGRES_DB','pedidosdb'), user=os.environ.get('POSTGRES_USER','postgres'), password=os.environ.get('POSTGRES_PASSWORD','postgres') ) def get_rabbit(): creds = pika.PlainCredentials(RABBIT_USER, RABBIT_PASS) conn = pika.BlockingConnection(pika.ConnectionParameters( host=RABBIT_HOST, credentials=creds, heartbeat=60)) ch = conn.channel() ch.exchange_declare(exchange='pedidos', exchange_type='direct', durable=True) ch.queue_declare(queue='pedidos.queue', durable=True) ch.queue_bind(queue='pedidos.queue', exchange='pedidos', routing_key='pedido') return conn, ch def init_db(): conn = get_db(); cur = conn.cursor() cur.execute(""" CREATE TABLE IF NOT EXISTS pedidos ( id VARCHAR(36) PRIMARY KEY, cliente VARCHAR(100) NOT NULL, total NUMERIC(10,2) DEFAULT 0, status VARCHAR(20) DEFAULT 'pendente', criado_em TIMESTAMP DEFAULT NOW() ); CREATE TABLE IF NOT EXISTS itens_pedido ( id SERIAL PRIMARY KEY, pedido_id VARCHAR(36) REFERENCES pedidos(id), produto VARCHAR(100) NOT NULL, qty INT NOT NULL, preco NUMERIC(10,2) NOT NULL ); """) conn.commit(); cur.close(); conn.close() @app.route('/order', methods=['POST']) def create_order(): data = request.get_json() or {} cliente = data.get('cliente','anonimo') itens = data.get('itens', []) if not itens: return jsonify({'error':'itens obrigatorio'}), 400 order_id = str(uuid.uuid4())[:8] payload = json.dumps({'order_id': order_id, 'cliente': cliente, 'itens': itens}) try: conn, ch = get_rabbit() ch.basic_publish( exchange='pedidos', routing_key='pedido', body=payload, properties=pika.BasicProperties(delivery_mode=2) # persistent ) conn.close() except Exception as e: return jsonify({'error': str(e)}), 500 return jsonify({'order_id': order_id, 'status': 'queued'}) @app.route('/health') def health(): return 'ok' if __name__ == '__main__': time.sleep(6) init_db() app.run(host='0.0.0.0', port=5000)
worker/worker.py — Consome com ACK manual
worker/worker.py
Python · pika ACK
import os, time, json import pika, psycopg2 RABBIT_HOST = os.environ.get('RABBIT_HOST','rabbitmq') RABBIT_USER = os.environ.get('RABBIT_USER','guest') RABBIT_PASS = os.environ.get('RABBIT_PASS','guest') def get_db(): return psycopg2.connect( host=os.environ.get('DB_HOST','db'), dbname=os.environ.get('POSTGRES_DB','pedidosdb'), user=os.environ.get('POSTGRES_USER','postgres'), password=os.environ.get('POSTGRES_PASSWORD','postgres') ) def processar(ch, method, properties, body): try: data = json.loads(body) order_id = data['order_id'] cliente = data['cliente'] itens = data['itens'] # Validação simulada de estoque for item in itens: if item.get('qty', 0) <= 0: print(f"NACK: qty inválida em {order_id}") ch.basic_nack(delivery_tag=method.delivery_tag, requeue=False) return total = sum(i.get('preco',0) * i.get('qty',1) for i in itens) conn = get_db(); cur = conn.cursor() cur.execute( "INSERT INTO pedidos (id, cliente, total, status) VALUES (%s,%s,%s,'processado')", (order_id, cliente, total) ) for item in itens: cur.execute( "INSERT INTO itens_pedido (pedido_id,produto,qty,preco) VALUES (%s,%s,%s,%s)", (order_id, item['produto'], item['qty'], item['preco']) ) conn.commit(); cur.close(); conn.close() # ACK somente após INSERT bem-sucedido ch.basic_ack(delivery_tag=method.delivery_tag) print(f"ACK: pedido {order_id} processado. Total: R${total:.2f}") except Exception as e: print(f"Erro: {e}") ch.basic_nack(delivery_tag=method.delivery_tag, requeue=True) def run(): creds = pika.PlainCredentials(RABBIT_USER, RABBIT_PASS) conn = pika.BlockingConnection(pika.ConnectionParameters( host=RABBIT_HOST, credentials=creds, heartbeat=60)) ch = conn.channel() ch.exchange_declare(exchange='pedidos', exchange_type='direct', durable=True) ch.queue_declare(queue='pedidos.queue', durable=True) ch.queue_bind(queue='pedidos.queue', exchange='pedidos', routing_key='pedido') ch.basic_qos(prefetch_count=1) # processa um por vez — fair dispatch ch.basic_consume(queue='pedidos.queue', on_message_callback=processar) print('Worker aguardando pedidos...') ch.start_consuming() while True: try: run() except Exception as e: print(f'Reconectando: {e}') time.sleep(5)

painel/app.py

import os, time import psycopg2 from flask import Flask, jsonify app = Flask(__name__) def get_db(): return psycopg2.connect( host=os.environ.get('DB_HOST','db'), dbname=os.environ.get('POSTGRES_DB','pedidosdb'), user=os.environ.get('POSTGRES_USER','postgres'), password=os.environ.get('POSTGRES_PASSWORD','postgres') ) @app.route('/') def index(): conn = get_db(); cur = conn.cursor() cur.execute(""" SELECT p.id, p.cliente, p.total, p.status, p.criado_em, COUNT(i.id) as num_itens FROM pedidos p LEFT JOIN itens_pedido i ON i.pedido_id = p.id GROUP BY p.id ORDER BY p.criado_em DESC LIMIT 50 """) rows = cur.fetchall(); cur.close(); conn.close() linhas = ''.join( f'<tr><td>{r[0]}</td><td>{r[1]}</td><td>R${r[2]:.2f}</td><td>{r[3]}</td><td>{r[4].strftime("%H:%M:%S")}</td><td>{r[5]}</td></tr>' for r in rows ) return f"""<!DOCTYPE html><html><head><meta charset="UTF-8"> <meta http-equiv="refresh" content="4"> <title>Painel de Pedidos</title> <style>body{{font-family:sans-serif;padding:2rem}}table{{border-collapse:collapse;width:100%}} th,td{{padding:8px 12px;border:1px solid #ddd}}th{{background:#f0f0f0}}</style></head> <body><h1>Painel de Pedidos</h1> <table><tr><th>ID</th><th>Cliente</th><th>Total</th><th>Status</th><th>Hora</th><th>Itens</th></tr> {linhas}</table></body></html>""" @app.route('/orders') def orders_json(): conn = get_db(); cur = conn.cursor() cur.execute("SELECT id, cliente, total, status FROM pedidos ORDER BY criado_em DESC LIMIT 20") rows = cur.fetchall(); cur.close(); conn.close() return jsonify([{'id':r[0],'cliente':r[1],'total':float(r[2]),'status':r[3]} for r in rows]) if __name__ == '__main__': time.sleep(5) app.run(host='0.0.0.0', port=3000)

Módulo 2 — Kubernetes / k3d

Worker — Deployment sem Service + livenessProbe

O worker não expõe porta. Adicionamos livenessProbe via exec: se o processo pika travar, o Kubernetes reinicia o pod automaticamente — destaque do App 03.

k8s/worker-deployment.yaml
Deployment + livenessProbe
apiVersion: apps/v1 kind: Deployment metadata: name: worker namespace: processador-pedidos spec: replicas: 1 selector: matchLabels: app: worker template: metadata: labels: app: worker spec: containers: - name: worker image: pedidos-worker:v1 imagePullPolicy: Never # sem ports — worker não recebe conexões envFrom: - secretRef: name: db-secret env: - name: RABBIT_HOST value: rabbitmq - name: DB_HOST value: db livenessProbe: # reinicia se o processo travar exec: command: ["python","-c","import pika; print('ok')"] initialDelaySeconds: 30 periodSeconds: 30 failureThreshold: 3
aplicar
kubectl apply -f k8s/worker-deployment.yaml
livenessProbe vs readinessProbe:
readinessProbe — controla se o pod recebe tráfego (usado na loja Flask).
livenessProbe — controla se o pod está "vivo"; se falhar N vezes, o pod é reiniciado. Ideal para workers que podem travar no meio de uma conexão pika.

Módulo 3 — Questões

Questões de avaliação

3 questões conceituais sobre RabbitMQ, ACK e K8s + 2 práticas de modificação de código.

  • Questão 1
    conceitual
    Qual a diferença entre Redis List e RabbitMQ para filas de mensagens? Em quais situações você escolheria um ou outro?
    💡 Redis List: simples, rápido, sem ACK nativo — se o worker crashar após BLPOP, a mensagem se perde. RabbitMQ: ACK/NACK explícito, mensagens ficam "unacked" até confirmação, reentrega automática em caso de falha, roteamento flexível via exchange. Use Redis para filas simples onde perda eventual é tolerável; RabbitMQ para operações críticas (pagamentos, pedidos).
  • Questão 2
    conceitual
    O que é ACK em mensageria e por que ele é importante? O que acontece se o worker enviar ACK antes de fazer o INSERT no Postgres?
    💡 ACK (acknowledgement) é a confirmação de que a mensagem foi processada com sucesso. Após ACK, o broker remove a mensagem. Se enviar ACK antes do INSERT e o banco falhar, a mensagem é perdida para sempre. Por isso o padrão correto é: processa completamente → só então ACK. Isso garante "at-least-once delivery".
  • Questão 3
    conceitual
    O que acontece com um pedido se o worker crashar no meio do processamento — após receber a mensagem do RabbitMQ mas antes de enviar ACK?
    💡 A mensagem fica no estado "unacked" no RabbitMQ. Quando a conexão TCP do worker cai (crash), o RabbitMQ detecta o fechamento da conexão e recoloca a mensagem na fila automaticamente (requeue). Um novo worker (ou o pod reiniciado) receberá a mensagem novamente. Isso pode causar processamento duplicado — daí a importância de idempotência no INSERT (upsert em vez de insert puro).
  • Questão 4
    prática
    Adicione o status "cancelado": implemente o endpoint POST /order/<id>/cancel na loja Flask que publica uma mensagem de cancelamento em uma fila cancelamentos.queue no RabbitMQ.
    💡 Crie uma exchange cancelamentos (direct) e uma fila cancelamentos.queue. O endpoint deve publicar {"order_id": id, "motivo": "cliente solicitou"}. Crie um segundo consumer no worker que também processa cancelamentos.queue e atualiza o status do pedido no Postgres para "cancelado".
  • Questão 5
    prática
    Adicione uma Dead Letter Queue (DLQ) para pedidos com erro de validação: em vez de descartar mensagens inválidas com NACK (requeue=False), envie-as para uma fila pedidos.dlq.
    💡 Configure a fila principal com o argumento x-dead-letter-exchange apontando para uma exchange dlx. Quando o worker envia NACK com requeue=False, o RabbitMQ automaticamente encaminha para a DLX → DLQ. Adicione um endpoint GET /dlq no painel que lista mensagens na dead letter queue para auditoria.
